Must follow the Agent Skills specification format (per research). Structure:

   - `name: default` (MUST match parent directory name per spec -- NOT "1nce-api" or "1NCE API")
   - `description:` 1-1024 chars describing the skill. Example: "Work with 1NCE IoT connectivity APIs. Covers authentication, SIM management, order management, and device operations. Use when integrating with 1NCE platform APIs."

2. **H1 title**: `# 1NCE API Integration`

3. **Authentication section** (per D-01): Step-by-step walkthrough:
   - State that all API calls use base URL `https://api.1nce.com/management-api`
   - Explain: user needs 1NCE Management Portal credentials with API access
   - Auth method: HTTP Basic Authentication to obtain Bearer token
   - Endpoint: `POST /oauth/token` with body `{"grant_type": "client_credentials"}`
   - Include curl example (per D-11):
     ```
     curl -X POST https://api.1nce.com/management-api/oauth/token \
       -u "your-username:your-password" \
       -H "Content-Type: application/json" \
       -d '{"grant_type": "client_credentials"}'
     ```
   - Response shape: `{"access_token": "...", "token_type": "bearer", "expires_in": 3600}`
   - Token expires in 3600 seconds (1 hour). Request a new token before expiry. Do NOT cache tokens beyond their expiry.
   - Usage: `Authorization: Bearer {access_token}` header on all subsequent requests

4. **Common Workflows section** (per D-02, D-09): 2-4 multi-step workflows. Analyze the API specs to pick the most useful. Recommended workflows from research:
   - **SIM Lifecycle Management**: List SIMs (`GET /v1/sims`) -> check individual SIM status (`GET /v1/sims/{iccid}`) -> get connectivity info (`GET /v1/sims/{iccid}/connectivity`) -> check usage/quota (`GET /v1/sims/{iccid}/quota/data`)
   - **Device Onboarding with 1NCE OS**: Register device -> configure PSK credentials -> set up cloud integration (AWS IoT, webhook)
   - **Connectivity Monitoring**: Get SIM connectivity info -> check data usage -> handle top-up if quota low (`POST /v1/sims/{iccid}/quota/top-up`)
   
   For each workflow: describe the sequence of API calls, which spec/endpoint is involved, and the logical flow. No full code snippets per workflow (per D-02) -- describe the sequence and which APIs are involved. Include 1-2 curl examples for key operations (per D-11).

5. **Gotchas and Best Practices section** (per D-03, D-04):
   - **Rate limiting**: Document any rate limiting behavior found in specs (check headers, response codes)
   - **Common error codes**: From authorization.json: `BadCredentials`, `AuthValidationError`, `BodyValidationError`, `UnsupportedContentTypeError`. Include HTTP status codes and what each means.
   - **CORS limitation**: Browser-based API calls will fail due to CORS restrictions. Use server-side proxy or backend calls instead.
   - **Token expiry handling**: Always check token age before requests. If expired, obtain a new token. Do not retry with an expired token.
   - **Base URL consistency**: All management API calls go to `https://api.1nce.com/management-api`. Do not omit the `/management-api` path prefix.
   - **Content-Type**: Always send `Content-Type: application/json` for POST/PUT requests.
   - **Pagination**: If list endpoints support pagination, document the pattern (check spec for query parameters like `page`, `pageSize`, `limit`).
   - Reference full API docs at `https://help.1nce.com` for complete endpoint details.

6. **Tone** (per D-10): Written FOR AI coding agents. Direct, imperative. "Use Bearer token in Authorization header." "Do not cache tokens beyond expiry." Structured for machine consumption. Assume the reader is an LLM helping a developer.

7. **Length constraint** (from research pitfall 3): Keep under 500 lines / ~3000 words. Focus on patterns and workflows, not exhaustive endpoint documentation.

Per D-06 and research Pattern 2, add a dedicated step to the deploy workflow that fixes the content-type for .md files in S3.

**In the `deploy-production` job**, add a new step named `Fix content-type for Markdown files` AFTER the "Sync remaining files with short cache" step (line 374) and BEFORE the "Invalidate CloudFront cache" step (line 376). This ordering is critical (research pitfall 5) -- the fix must run AFTER both S3 sync steps so it does not get overwritten.

The new step:
```yaml
      - name: Fix content-type for Markdown files
        run: |
          aws s3 cp s3://1nce-developer-hub-prod/ s3://1nce-developer-hub-prod/ \
            --recursive \
            --exclude "*" \
            --include "*.md" \
            --content-type "text/markdown; charset=utf-8" \
            --metadata-directive REPLACE \
            --cache-control "public, max-age=600, must-revalidate"
```

This copies .md files in-place within the same bucket, overriding only the metadata (content-type and cache-control). It uses `--metadata-directive REPLACE` to ensure the new content-type takes effect. The `charset=utf-8` suffix prevents UTF-8 rendering issues (research pitfall 4).

**In the `deploy-preview` job**, add the same fix AFTER the "Deploy to preview S3" step (line 58). Adjust the bucket path for preview:
```yaml
      - name: Fix content-type for Markdown files
        run: |
          aws s3 cp s3://1nce-developer-hub-preview/pr-${{ github.event.number }}/ s3://1nce-developer-hub-preview/pr-${{ github.event.number }}/ \
            --recursive \
            --exclude "*" \
            --include "*.md" \
            --content-type "text/markdown; charset=utf-8" \
            --metadata-directive REPLACE \
            --cache-control "public, max-age=600, must-revalidate"
```

Do NOT modify any existing steps. Only add the two new steps.

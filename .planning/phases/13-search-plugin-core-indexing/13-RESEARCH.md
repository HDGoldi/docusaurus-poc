# Phase 13: Search Plugin & Core Indexing - Research

**Researched:** 2026-04-05
**Domain:** Client-side full-text search integration for Docusaurus 3.9.2
**Confidence:** HIGH

## Summary

Phase 13 adds client-side full-text search to a Docusaurus 3.9.2 site serving 423 pages (166 documentation MDX + 126 API MDX across two docs plugin instances). The implementation is a single npm package (`@easyops-cn/docusaurus-search-local` v0.55.1) registered as a Docusaurus theme, requiring approximately 15 lines of configuration in `docusaurus.config.ts` and zero changes to existing components, CI/CD, or infrastructure.

The two critical risks are: (1) the dual docs-instance setup requires explicit `docsRouteBasePath: ['/docs', '/api']` array configuration -- omitting this causes all 126 API pages to be silently invisible in search; and (2) the project's Rspack bundler (`experimental_faster: true`) has zero reported community usage with this plugin. Both risks are diagnosable on first build and have pre-planned fallback paths (D-01 through D-03).

Per the CONTEXT.md decisions, Phase 13 ships with plugin defaults only -- no custom CSS, no branding overrides, no search context filtering. Redirect stub pages (80+) are excluded via `ignoreFiles`. All UI polish, branding, and context filtering are deferred to Phase 14.

**Primary recommendation:** Install `@easyops-cn/docusaurus-search-local` v0.55.1, configure dual-instance indexing, build, and validate API pages appear in the search index before proceeding with any other work.

## Architecture Patterns

### Recommended Configuration

```typescript
// docusaurus.config.ts -- add to existing themes array
themes: [
  'docusaurus-theme-openapi-docs',
  ['@easyops-cn/docusaurus-search-local', {
    indexDocs: true,
    indexBlog: false,
    indexPages: false,
    docsRouteBasePath: ['/docs', '/api'],       // CRITICAL: both docs instances
    docsDir: ['docs/documentation', 'docs/api'], // CRITICAL: matching source dirs
    language: ['en'],
    hashed: 'filename',                          // Cache-friendly for CloudFront
    searchBarShortcutHint: true,                 // Show Cmd+K hint in search bar
    searchBarPosition: 'right',                  // Right side of navbar
    searchResultLimits: 8,                       // Plugin default, adequate for 423 pages
    explicitSearchResultPath: true,              // Show breadcrumb paths to distinguish docs vs API
    // ignoreFiles: [<redirect stub patterns>],  // Claude's discretion -- see below
  }],
],
```
[CITED: .planning/research/ARCHITECTURE.md, .planning/research/STACK.md]

### Pattern 1: Theme-Based Search Integration

**What:** Search plugins register as Docusaurus "themes" (not "plugins") because they provide UI components (SearchBar) that fill Docusaurus's built-in navbar component slot.

**When to use:** Always -- this is the only correct integration method.

**Key detail:** The `themes` array in `docusaurus.config.ts` already contains `docusaurus-theme-openapi-docs`. The search theme is added alongside it. No conflicts -- they provide different components (ApiItem vs SearchBar). [CITED: .planning/research/ARCHITECTURE.md]

### Pattern 2: PostBuild Index Generation

**What:** The search index is built AFTER the full Docusaurus build completes, by scanning static HTML output in `build/`. This means API pages generated by openapi-docs are already compiled to HTML when indexing runs.

**Key implication:** Search is NOT available in `docusaurus start` (dev mode). Must use `npm run build && npm run serve` to test search. [CITED: .planning/research/ARCHITECTURE.md]

### Pattern 3: Redirect Stub Exclusion via ignoreFiles

**What:** The project has 80+ redirect rules in `plugin-client-redirects`. Each generates an HTML stub page in `build/`. These stubs would pollute search results with duplicate/empty entries.

**Implementation:** The `ignoreFiles` option accepts glob patterns to exclude files from indexing. The redirect stubs are generated at build time in paths like `build/dev-hub/**`, `build/blueprints-examples/**`, `build/platform/**`, etc. The exact pattern should be determined by inspecting `build/` output after first build, but based on the redirect config in `docusaurus.config.ts`, the following paths should be excluded:

```typescript
ignoreFiles: [
  // Old ReadMe paths that generate redirect stubs
  /dev-hub\//,
  /blueprints-examples\/(?!examples-)/,  // Top-level blueprint redirects
  /^platform\//,                          // Old /platform/ prefix redirects
  /^starting-guide\//,                    // Old starting-guide redirect
  /\/index$/,                             // Old /index suffix redirects
],
```

**Note (Claude's discretion):** The exact patterns should be verified after first build by listing HTML files in `build/` that are redirect stubs. A pragmatic approach is to build first, then inspect and add patterns in a follow-up commit. [ASSUMED]

### Anti-Patterns to Avoid

- **Adding search to `plugins` array instead of `themes`:** The package provides a React component (SearchBar). Plugins cannot provide theme components. Search bar will not appear. [CITED: .planning/research/ARCHITECTURE.md]
- **Forgetting `/api` in `docsRouteBasePath`:** Default is `'/docs'` (string). API pages at `/api/` will be silently excluded from search. Must use array. [CITED: .planning/research/ARCHITECTURE.md]
- **Testing search in dev mode:** `npm start` does not run postBuild hooks. No index exists. Must use `npm run build && npm run serve`. [CITED: .planning/research/ARCHITECTURE.md]
- **Swizzling SearchBar prematurely:** CSS customization via `custom.css` is sufficient. Swizzling creates maintenance burden and blocks upstream fixes. Per D-10, use plugin defaults for Phase 13. [CITED: .planning/research/ARCHITECTURE.md]

## Common Pitfalls

### Pitfall 1: API Pages Silently Not Indexed

**What goes wrong:** After installing the search plugin, the 126 API endpoint pages at `/api/` do not appear in search results. Users search for "Obtain Access Token" or "Get SIM Details" and get zero hits.
**Why it happens:** Default `docsRouteBasePath` is `'/docs'` (single string). Without the array config `['/docs', '/api']`, the plugin only scans pages under `/docs/`.
**How to avoid:** Configure `docsRouteBasePath: ['/docs', '/api']` and `docsDir: ['docs/documentation', 'docs/api']`. After first build, grep the raw search-index JSON for known API titles. This is a hard go/no-go gate.
**Warning signs:** Search index file is suspiciously small (<1 MB for 423 pages). Results only show `/docs/` paths.
[CITED: .planning/research/PITFALLS.md, GitHub issue #550]

### Pitfall 2: Rspack Bundler Incompatibility

**What goes wrong:** Build fails or search renders but produces no results when `experimental_faster: true` is enabled.
**Why it happens:** Zero community reports of this plugin running with Rspack as of April 2026. The plugin uses `configureWebpack` which may not merge correctly under Rspack. The existing `polyfillNodeModules` plugin also uses `configureWebpack` -- multiple returns may interact poorly.
**How to avoid:** Run `npm run build` immediately after install. If search does not work, disable `experimental_faster` per D-01/D-02. Do not debug -- ship without Rspack. Build time increase (~30-60s) is acceptable per D-03.
**Warning signs:** Build warnings about unsupported loader options. Search bar shows "Loading..." indefinitely. Console errors about missing modules or worker initialization.
[CITED: .planning/research/PITFALLS.md, zero GitHub results for "rspack" in plugin repo]

### Pitfall 3: Oversized Search Index

**What goes wrong:** Index file exceeds 5 MB uncompressed. 126 API pages with multi-language code samples contribute disproportionately to index weight.
**Why it happens:** lunr.js indexes full text content. OpenAPI pages contain repetitive JSON schemas and code snippets in 5+ languages per endpoint.
**How to avoid:** Measure index size after first build (D-08, D-09). Exclude redirect stubs via `ignoreFiles` (D-07). Further optimization deferred to Phase 14.
**Warning signs:** `build/search-index-*.json` exceeds 5 MB. Mobile users report search spinning or freezing.
[CITED: .planning/research/PITFALLS.md, GitHub issue #549]

### Pitfall 4: Redirect Stub Pages Polluting Search Results

**What goes wrong:** 80+ redirect HTML stubs appear in search results with empty/duplicate content.
**Why it happens:** `plugin-client-redirects` generates HTML stub pages in `build/` that the search indexer scans like any other page.
**How to avoid:** Add `ignoreFiles` patterns matching redirect paths. Verify after build by searching for old ReadMe slugs like "dev-hub" or "introduction-welcome".
[CITED: .planning/research/PITFALLS.md, docusaurus.config.ts redirect list]

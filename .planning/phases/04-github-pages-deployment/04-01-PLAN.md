  <action>
Create `.github/workflows/gh-pages.yml` per D-01 and D-03. This is a NEW file — does not modify `deploy.yml`.

Create the file with this exact content:

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci

      - name: Build for GitHub Pages
        env:
          DEPLOY_TARGET: gh-pages
        run: npm run build

      - uses: actions/upload-pages-artifact@v3
        with:
          path: build

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

Key details:
- `permissions`: `pages: write` and `id-token: write` required for OIDC-based Pages deployment. `contents: read` for checkout.
- `concurrency`: `cancel-in-progress: true` prevents queued stale deploys.
- `DEPLOY_TARGET: gh-pages` env var triggers the config switching from Task 1.
- `actions/upload-pages-artifact@v3` + `actions/deploy-pages@v4` is the modern Pages deployment approach (no gh-pages branch needed).
- `environment: github-pages` is auto-created by GitHub when Pages is enabled.
- Node 20 and npm cache match the existing deploy.yml pattern.

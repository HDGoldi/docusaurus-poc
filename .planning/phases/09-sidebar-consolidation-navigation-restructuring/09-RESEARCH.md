# Phase 9: Sidebar Consolidation & Navigation Restructuring - Research

**Researched:** 2026-04-02
**Domain:** Docusaurus multi-instance plugin consolidation, sidebar ordering, client-side redirects
**Confidence:** HIGH

## Summary

This phase consolidates 3 separate Docusaurus `@docusaurus/plugin-content-docs` instances (platform, blueprints, terms) into the single main Documentation instance served by the preset-classic docs plugin. The content directories are physically moved under `docs/documentation/`, the 3 plugin entries and their sidebar files are removed, `_category_.json` files enforce the 11-section sidebar order, and `@docusaurus/plugin-client-redirects` handles old URL redirects.

The codebase is well-structured for this change. All 4 sidebar files already use `autogenerated` mode, meaning the sidebar rebuilds automatically from directory structure. The existing `_category_.json` files in `docs/documentation/` already have correct position values (1, 3, 4, 5, 7, 9) matching the target 11-section order. The 3 directories being moved in from `docs/platform/` also already have `_category_.json` with correct positions (2, 6, 8). Only the blueprints-examples category needs its label updated (currently "Blueprints Examples", needs "Blueprints & Examples") and the terms standalone doc needs a `sidebar_position: 11` frontmatter addition.

**Primary recommendation:** Move content directories first, update `_category_.json` files, then modify `docusaurus.config.ts` and add the redirects plugin -- doing config and content moves in separate logical steps prevents broken intermediate states.

### Pattern 4: Client Redirects Plugin Configuration
**What:** `@docusaurus/plugin-client-redirects` creates client-side redirects (HTML meta refresh pages at old URLs).
**Configuration:**
```typescript
// In docusaurus.config.ts plugins array
['@docusaurus/plugin-client-redirects', {
  redirects: [
    // Platform redirects
    { from: '/platform/1nce-portal/portal-dashboard', to: '/docs/1nce-portal/portal-dashboard' },
    // ... etc
  ],
}],
```
**Important:** The plugin uses `from` (old path) and `to` (new path). Paths should NOT include trailing slashes in the config -- the plugin handles trailingSlash configuration automatically. However, since this site has `trailingSlash: true`, verify redirect behavior includes trailing slash variants.

### Anti-Patterns to Avoid
- **Moving files one-by-one with individual git commits:** Move all files in a single operation to avoid broken intermediate states where config references directories that don't exist.
- **Editing `sidebars/documentation.ts` to manually list items:** The autogenerated sidebar pattern is already correct -- `_category_.json` handles ordering. Do not switch to manual sidebar definition.
- **Forgetting to remove the old empty directories:** After moving content, the empty `docs/platform/`, `docs/blueprints/`, `docs/terms/` directories should be removed (git will handle this automatically since it tracks files not directories).

## Common Pitfalls

### Pitfall 1: Plugin Instance ID References in Navbar
**What goes wrong:** Removing plugin instances but forgetting to remove navbar items that reference `docsPluginId: 'platform'` etc. causes build errors.
**Why it happens:** Navbar items reference plugin IDs, and Docusaurus validates these at build time.
**How to avoid:** Remove both the plugin entries AND the corresponding navbar items in the same config change.
**Warning signs:** Build error: "Plugin with ID 'platform' not found"

### Pitfall 2: Duplicate Doc IDs After Merge
**What goes wrong:** If two docs across different plugin instances share the same `id` (from frontmatter or filename), merging them into one instance causes a collision.
**Why it happens:** Separate plugin instances have separate ID namespaces.
**How to avoid:** Since all files are in separate subdirectories with unique names, this is LOW risk. But verify by running `npm run build` after the merge.
**Warning signs:** Build error about duplicate document IDs.

### Pitfall 3: Redirect Paths Must Match routeBasePath
**What goes wrong:** Redirects don't work because `from` paths don't match how Docusaurus served the old URLs.
**Why it happens:** The old plugin instances used `routeBasePath: '/platform'`, `/blueprints'`, '/terms'`. The `from` paths in redirects must match these exactly.
**How to avoid:** Old URLs follow pattern: `/{routeBasePath}/{relative-path-in-docs-dir}`. Map systematically.
**Warning signs:** 404 on old URLs after deployment.

### Pitfall 4: trailingSlash Interaction with Redirects
**What goes wrong:** Site has `trailingSlash: true` but redirect `from` paths may need trailing slashes too.
**Why it happens:** `@docusaurus/plugin-client-redirects` respects the site's `trailingSlash` setting when generating redirect pages.
**How to avoid:** Write `from` paths WITHOUT trailing slashes -- the plugin adds them automatically based on site config.
**Warning signs:** Redirect pages not generated at expected paths.

### Pitfall 5: Index Pages in Moved Directories
**What goes wrong:** Categories with `index.md` files generate URLs at the category root (e.g., `/platform/1nce-os/1nce-os-admin-logs/` from `index.md`). These need redirects too.
**Why it happens:** `index.md` files serve as category landing pages and have their own URL.
**How to avoid:** Include index page paths in the redirect map. For a directory like `platform/1nce-os/1nce-os-admin-logs/index.md`, the old URL is `/platform/1nce-os/1nce-os-admin-logs/` and new URL is `/docs/1nce-os/1nce-os-admin-logs/`.

## Redirect Map

### URL Transformation Rules

**Platform content:**
- Old: `/platform/{subdir}/{path}` -> New: `/docs/{subdir}/{path}`
- Example: `/platform/1nce-os/1nce-os-admin-logs/admin-logs-api` -> `/docs/1nce-os/1nce-os-admin-logs/admin-logs-api`
- 3 top-level subdirs: `1nce-os/`, `1nce-portal/`, `platform-services/`
- **71 markdown files** to generate redirects for

**Blueprints content:**
- Old: `/blueprints/blueprints-examples/{path}` -> New: `/docs/blueprints-examples/{path}`
- Old: `/blueprints/{device-doc}` -> New: `/docs/blueprints-examples/{device-doc}` (if device docs move into blueprints-examples)
- **62 markdown files** to generate redirects for

**Terms content:**
- Old: `/terms/terms-abbreviations` -> New: `/docs/terms-abbreviations`
- **1 file** to redirect

**Total: ~134 redirect entries**

### Programmatic Generation Approach
Rather than hand-writing 134 redirect entries, use `createRedirects` function in the plugin config:
```typescript
['@docusaurus/plugin-client-redirects', {
  createRedirects(existingPath) {
    // Platform content: /docs/1nce-os/* was /platform/1nce-os/*
    if (existingPath.includes('/docs/1nce-os/') ||
        existingPath.includes('/docs/1nce-portal/') ||
        existingPath.includes('/docs/platform-services/')) {
      const section = existingPath.match(/\/docs\/(1nce-os|1nce-portal|platform-services)\//)?.[1];
      if (section) {
        return [existingPath.replace('/docs/', '/platform/')];
      }
    }
    // Blueprints content: /docs/blueprints-examples/* was /blueprints/blueprints-examples/*
    if (existingPath.includes('/docs/blueprints-examples/')) {
      return [existingPath.replace('/docs/blueprints-examples/', '/blueprints/blueprints-examples/')];
    }
    // Terms: /docs/terms-abbreviations was /terms/terms-abbreviations
    if (existingPath.includes('/docs/terms-abbreviations')) {
      return ['/terms/terms-abbreviations'];
    }
    return undefined;
  },
}],
```

This is cleaner than 134 hardcoded entries and automatically handles any future content additions.

### Edge Cases for Blueprints Top-Level Files
The 6 top-level `.md` files in `docs/blueprints/` (quectel-bg95-m3.md, recipes.md, etc.) currently serve at `/blueprints/quectel-bg95-m3`, `/blueprints/recipes`, etc. If moved into `docs/documentation/blueprints-examples/`, they will serve at `/docs/blueprints-examples/quectel-bg95-m3`. The `createRedirects` function needs a rule for these:
```typescript
// Top-level blueprints files that moved into blueprints-examples
if (existingPath.includes('/docs/blueprints-examples/') &&
    !existingPath.includes('/docs/blueprints-examples/blueprints-examples/')) {
  // Also redirect from /blueprints/{slug} for top-level files
  const slug = existingPath.split('/docs/blueprints-examples/')[1];
  if (['quectel-bg95-m3', 'quectel-ec25-ec21', 'recipes', 'sara-r410m', 'sim7000g', 'simcom-7020g-simcom800l'].some(s => slug?.startsWith(s))) {
    return [
      existingPath.replace('/docs/blueprints-examples/', '/blueprints/blueprints-examples/'),
      existingPath.replace('/docs/blueprints-examples/', '/blueprints/')
    ];
  }
  return [existingPath.replace('/docs/blueprints-examples/', '/blueprints/blueprints-examples/')];
}
```

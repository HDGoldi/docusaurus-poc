<task type="checkpoint:decision" gate="blocking">
  <name>Task 1: User gate -- confirm context filtering behavior (D-06/D-07/D-08)</name>
  <files>docusaurus.config.ts</files>
  <decision>D-06/D-07/D-08 assumed the search plugin would create tab-based filtering ("All" | "Docs" | "API") above search results. Research confirmed this is NOT what the plugin provides. The user must confirm the acceptable approach before implementation proceeds.</decision>
  <context>
The `searchContextByPaths` plugin option creates separate search indexes per path and auto-detects context from the current page URL. It does NOT render tabs, buttons, or a dropdown for switching contexts.

Actual behavior:
- On /docs/* pages: only documentation results appear
- On /api/* pages: only API results appear
- On the homepage or other pages: all results appear (via useAllContextsWithNoSearchContext: true)
- A footer link in results allows bridging to "See all results"

This is functionally different from the tab-based UI described in D-06/D-07/D-08. Building actual tabs would require swizzling the SearchBar React component -- significant custom work with high upgrade risk.
  </context>
  <options>
    <option id="option-a">
      <name>Accept auto-detection (recommended)</name>
      <pros>Zero custom code. Plugin-native behavior. Users on /docs pages see docs results, users on /api pages see API results, homepage shows all. "See all results" footer link available. No upgrade risk.</pros>
      <cons>No visible tabs in the UI. Users cannot manually switch between "Docs" and "API" contexts from a single page. Filtering is implicit based on URL, not explicit user choice.</cons>
    </option>
    <option id="option-b">
      <name>Build custom tab UI via SearchBar swizzle</name>
      <pros>Matches the original D-06/D-07/D-08 tab vision exactly. Users can manually switch between "All", "Docs", "API" regardless of current page.</pros>
      <cons>Requires swizzling SearchBar component (CSS Module class names, upgrade burden). Significant custom React development. Goes against research "Don't swizzle" guidance. Increases maintenance risk on plugin updates.</cons>
    </option>
    <option id="option-c">
      <name>Defer context filtering entirely</name>
      <pros>Simplest. All searches return all results everywhere.</pros>
      <cons>Does not implement D-06/D-07/D-08 in any form. Mixed docs/API results on all pages.</cons>
    </option>
  </options>
  <action>Present the discrepancy between D-06/D-07/D-08 (tab-based filtering) and what the plugin actually provides (URL auto-detection). Block execution until user selects an option.</action>
  <verify>User has selected option-a, option-b, or option-c</verify>
  <done>User decision on context filtering approach recorded; subsequent tasks implement accordingly</done>

- D-04: Mark elements get rgba(41,171,226,0.15) background + bold + teal color
- D-05: article mark styled for target page highlights from mark.js
- UI-02: Darkened backdrop behind overlay
- UI-05: Light-only mode enforced via [data-theme='dark'] overrides
- UI-07: Mark element styling for result highlighting

NOTE: The backdrop implementation uses a ::before pseudo-element approach. If the plugin's DOM structure doesn't match this selector (class names are CSS Modules, so they include hashes), the executor should inspect the actual rendered DOM with `npm run build && npm run serve` and adjust the selector. The fallback approach is to target the body element when search is active via `body.search-active::before` if the plugin adds such a class.

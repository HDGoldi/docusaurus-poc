## Target Architecture

### Goal

Merge `platform` and `blueprints` content into the main `documentation` docs instance so they appear in a single "Documentation" sidebar matching the original ReadMe.com hub. Keep `terms` and `api` as separate plugins.

### Why This Structure

The original ReadMe.com hub had a single "Documentation" tab with all content (Portal, Platform Services, 1NCE OS, Blueprints) in one sidebar. The v1.0 migration split them into separate Docusaurus plugin instances for expedience, but this created a navigation mismatch. v1.2 restores the original UX.

├── preset-classic.docs          <- id: "default"
│   path: docs/documentation/    <- NOW contains all merged content
│   routeBasePath: /docs
│   sidebar: sidebars/documentation.ts (autogenerated with _category_.json positions)

## Architectural Patterns

### Pattern 1: Autogenerated Sidebar with _category_.json Position Control

**What:** Keep the `autogenerated` sidebar directive in `documentation.ts` but rely on `_category_.json` `position` fields in each directory to control the exact ordering of categories in the merged sidebar.

**When to use:** When you need precise sidebar ordering but want new pages to appear automatically without manual sidebar edits.

**Trade-offs:**
- Pro: Autogenerated sidebars adapt when files are added/removed -- zero sidebar maintenance.
- Pro: `_category_.json` position values give deterministic ordering.
- Con: Must set position on every top-level category to avoid alphabetical fallback.
- Con: Position numbers are only visible by inspecting individual `_category_.json` files, not in one place.

**Implementation:**

```typescript
// sidebars/documentation.ts -- keep simple, rely on _category_.json
import type {SidebarsConfig} from '@docusaurus/plugin-content-docs';

const sidebars: SidebarsConfig = {
  docsSidebar: [{type: 'autogenerated', dirName: '.'}],
};

export default sidebars;
```

```json
// docs/documentation/introduction/_category_.json
{ "label": "Introduction", "position": 1 }

// docs/documentation/1nce-portal/_category_.json
{ "label": "1NCE Portal", "position": 2 }

// docs/documentation/sim-cards/_category_.json
{ "label": "SIM Cards", "position": 3 }
```

**Target sidebar order (matching original ReadMe.com hub):**

### Pattern 2: Client-Side Redirects for URL Preservation

**What:** Use `@docusaurus/plugin-client-redirects` to redirect old `/platform/*` and `/blueprints/*` URLs to new `/docs/*` URLs. Prevents broken bookmarks and external links.

**When to use:** Whenever you change URL structure for existing published content.

**Trade-offs:**
- Pro: Zero infrastructure changes (no CloudFront Function edits needed).
- Pro: Generated as static HTML files during build -- no runtime cost.
- Con: Client-side only -- not a 301 for SEO. Acceptable for a docs site.
- Con: Generates extra HTML files in build output. Negligible for ~50 redirects.

**Implementation:**

```bash
npm install @docusaurus/plugin-client-redirects
```

```typescript
// In docusaurus.config.ts plugins array
['@docusaurus/plugin-client-redirects', {
  createRedirects(existingPath) {
    // Platform content moved from /platform/ to /docs/
    if (existingPath.startsWith('/docs/1nce-portal/') ||
        existingPath.startsWith('/docs/platform-services/') ||
        existingPath.startsWith('/docs/1nce-os/')) {
      return [existingPath.replace('/docs/', '/platform/')];
    }
    // Blueprints content moved from /blueprints/ to /docs/
    if (existingPath.startsWith('/docs/blueprints-examples/')) {
      return [existingPath.replace('/docs/', '/blueprints/')];
    }
    // Loose blueprint files moved from /blueprints/ to /docs/additional-resources/
    if (existingPath.startsWith('/docs/additional-resources/')) {
      const filename = existingPath.replace('/docs/additional-resources/', '');
      return [`/blueprints/${filename}`];
    }
    return [];
  },
}],
```

**Preferred approach:** Use the `createRedirects` function. It auto-generates redirects for any matching page without maintaining a manual list. Less error-prone as content evolves.

### Pattern 3: Handling Loose Files from Blueprints

**What:** The `docs/blueprints/` directory has both a `blueprints-examples/` subdirectory and 6 loose `.md` files at the root (`quectel-bg95-m3.md`, `recipes.md`, etc.). These loose files need a category in the merged structure.

**Recommendation:** Create `docs/documentation/additional-resources/` with a `_category_.json` and move the loose files there. This matches the "Additional Resources" category visible in the reference `sidebars.ts` at the project root.

```json
// docs/documentation/additional-resources/_category_.json
{
  "label": "Additional Resources",
  "position": 11
}
```

## Execution Order (Dependency-Driven)

The migration has a strict execution order. Recommended sequence:

### Step 1: Update docusaurus.config.ts (Config First)

Remove the `platform` and `blueprints` plugin entries and their navbar items BEFORE moving files. This prevents the build from trying to read from both old and new paths simultaneously.

Changes:
- Delete the `platform` plugin-content-docs entry from `plugins[]`
- Delete the `blueprints` plugin-content-docs entry from `plugins[]`
- Delete the "1NCE Platform" and "Blueprints & Examples" navbar items
- Add external navbar links (1NCE Home, Shop, Portal)

At this point the build would fail (removed plugins reference content that still exists at old paths). That is expected -- proceed immediately to Step 2.

### Step 2: Move Content Files

```bash
# Use git mv to preserve history
git mv docs/platform/1nce-portal docs/documentation/1nce-portal
git mv docs/platform/platform-services docs/documentation/platform-services
git mv docs/platform/1nce-os docs/documentation/1nce-os

mkdir -p docs/documentation/additional-resources
git mv docs/blueprints/blueprints-examples docs/documentation/blueprints-examples
git mv docs/blueprints/quectel-bg95-m3.md docs/documentation/additional-resources/
git mv docs/blueprints/quectel-ec25-ec21.md docs/documentation/additional-resources/
git mv docs/blueprints/recipes.md docs/documentation/additional-resources/
git mv docs/blueprints/sara-r410m.md docs/documentation/additional-resources/
git mv docs/blueprints/sim7000g.md docs/documentation/additional-resources/
git mv docs/blueprints/simcom-7020g-simcom800l.md docs/documentation/additional-resources/
```

### Step 3: Update _category_.json Positions

Set position values on all top-level categories in `docs/documentation/` to match the target ordering (see Pattern 1 above). Create `additional-resources/_category_.json`.

### Step 4: Install and Configure Client Redirects

```bash
npm install @docusaurus/plugin-client-redirects
```

Add the plugin to `docusaurus.config.ts` with the `createRedirects` function (see Pattern 2 above).

### Step 5: Clean Up

- Delete empty `docs/platform/` directory (should be empty after moves)
- Delete empty `docs/blueprints/` directory (should be empty after moves)
- Delete `sidebars/platform.ts`
- Delete `sidebars/blueprints.ts`

### Step 6: Build and Verify

Run `npm run build` and verify:
- All pages render at `/docs/*` URLs
- Sidebar order matches original hub (check against root `sidebars.ts` reference)
- Old `/platform/*` and `/blueprints/*` URLs produce redirect HTML files
- No broken links (use `onBrokenLinks: 'throw'` temporarily for strict checking)
- API Explorer unaffected at `/api/*`
- Terms unaffected at `/terms/*`
- Navbar shows correct items (Documentation, API Explorer, Terms, external links)

## Anti-Patterns

### Anti-Pattern 1: Using Multiple `path` Entries in One Plugin

**What people do:** Try to configure a single docs plugin to read from multiple directories (`docs/documentation/` AND `docs/platform/`).
**Why it is wrong:** Docusaurus `plugin-content-docs` accepts exactly one `path` string. There is no multi-path option.
**Do this instead:** Physically move all content into one directory tree.

### Anti-Pattern 2: Writing a Massive Explicit Sidebar

**What people do:** Write a hand-crafted sidebar object listing every single page (like the reference `sidebars.ts` at the project root with 700 lines).
**Why it is wrong:** Becomes a maintenance burden. Every new page requires a sidebar edit. Easy to miss pages or create stale entries that break the build.
**Do this instead:** Use `autogenerated` with `_category_.json` position fields. New pages appear automatically.

### Anti-Pattern 3: Server-Side Redirects for URL Changes

**What people do:** Add CloudFront Function rules to handle `/platform/*` to `/docs/*` redirects at the CDN level.
**Why it is wrong:** CloudFront Functions have a 10KB code size limit. Adding dozens of redirect rules bloats the function. Also requires infrastructure changes and separate deployment.
**Do this instead:** Use `@docusaurus/plugin-client-redirects`. It generates static HTML files during build. Zero infrastructure changes.

### Anti-Pattern 4: Symlinks Instead of File Moves

**What people do:** Create symlinks from `docs/documentation/1nce-os` to `docs/platform/1nce-os` to avoid moving files.
**Why it is wrong:** Docusaurus does not reliably follow symlinks during build. Creates confusing git history. Breaks on Windows.
**Do this instead:** `git mv` the files. Clean history, clean build.

### Anti-Pattern 5: Keeping Unused Plugin Instances

**What people do:** Leave the `platform` and `blueprints` plugin entries in config with empty paths or no content, thinking they are harmless.
**Why it is wrong:** Empty docs plugins still initialize during build, adding startup time and potential warnings. Unused sidebar files create confusion.
**Do this instead:** Remove plugin entries entirely. Delete their sidebar files.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Broken internal links between moved pages | MEDIUM | LOW | Links within platform content use relative paths -- these still work since directory structure is preserved. Run build with `onBrokenLinks: 'throw'` to catch any issues. |
| Frontmatter `slug` overrides causing URL collisions | LOW | MEDIUM | Search all `.md` files for `slug:` in frontmatter before merging. If any exist, verify they do not collide after the merge. |
| External links to `/platform/*` breaking | HIGH | MEDIUM | Fully mitigated by `plugin-client-redirects`. Test all redirect paths after build. |
| Build time change | LOW | LOW | Going from 4 docs plugins to 2 should slightly reduce build time (fewer plugin instances). |
| `_category_.json` from platform conflicting with documentation | LOW | LOW | Platform subdirectories have their own `_category_.json` files with internal positions. These control sub-category ordering within their parent and do not conflict with top-level positions. Only top-level positions need careful assignment. |

Not applicable -- this is a static site restructuring with no runtime implications. Build output size increases negligibly (redirect HTML files are a few KB each).

# Project Research Summary

**Project:** 1NCE Developer Hub — v1.4 Client-Side Search
**Domain:** Client-side full-text search integration for Docusaurus 3.9.2
**Researched:** 2026-04-05
**Confidence:** HIGH (stack, features, architecture), MEDIUM-HIGH (pitfalls), MEDIUM (Rspack compatibility)

## Executive Summary

This phase adds client-side full-text search to a Docusaurus 3.9.2 documentation site serving ~423 pages: 298 general documentation pages (at `/docs/`) and 125 API Explorer pages (at `/api/`) generated by `docusaurus-plugin-openapi-docs`. The recommended approach is `@easyops-cn/docusaurus-search-local` v0.55.1 — the only actively maintained Docusaurus 3.x search plugin with explicit support for multi-docs-instance configurations. At 195K weekly downloads and a February 2026 release, it is the clear community standard. The plugin installs as a Docusaurus theme, auto-injects a search bar into the navbar, and generates a lunr.js inverted index during the post-build step — requiring no infrastructure changes beyond what is already deployed.

The recommended approach trades Algolia's server-side relevance for zero cost and offline capability. For a 423-page site with weekly-or-less update cadence, this tradeoff is strongly favorable. All search happens client-side in a Web Worker against an index generated at build time — no external service dependencies, no API keys, and no recurring costs. The index updates automatically on every deploy via the existing GitHub Actions / S3 / CloudFront pipeline without any CI/CD changes.

The two critical risks are: (1) the dual docs-instance setup requires explicit `docsRouteBasePath: ['/docs', '/api']` array configuration — omitting this causes all 125 API pages to be silently invisible in search; and (2) the project's Rspack bundler (`experimental_faster: true`) has zero reported prior art with this plugin, making it an unknown compatibility risk that must be verified on first build. Both risks are diagnosable immediately after install and have straightforward recovery paths.

## Key Findings

### Recommended Stack

The stack for this phase is narrow and well-defined. One package satisfies the entire search requirement with no additional dependencies. The plugin bundles lunr.js, mark.js, cheerio, and its own autocomplete UI — nothing else needs to be installed. Integration is a single entry in the `themes` array of `docusaurus.config.ts` with approximately 10 configuration options.

**Core technologies:**
- `@easyops-cn/docusaurus-search-local` v0.55.1: client-side full-text search theme — only plugin with explicit array support for `docsRouteBasePath`/`docsDir`, covering the dual docs-instance requirement
- lunr.js (bundled): inverted index search engine — build-time indexing, runtime querying in a Web Worker; no server required
- CloudFront compression for `application/json` (existing infrastructure): gzip/brotli on the search index JSON — critical for keeping the ~1-5 MB index file performant on CDN delivery

**What NOT to add:**
- `@cmfcmf/docusaurus-search-local`: bundles unused Algolia client libraries; no documented multi-instance support
- `docusaurus-lunr-search`: stale (15 months since last release); no multi-instance array config
- Algolia DocSearch: explicitly out of scope; requires application/approval; paid at commercial scale
- Any server-side search engine (Typesense, Meilisearch): overkill for 423 pages; requires hosting infrastructure

### Expected Features

**Must have (table stakes — v1.4 launch):**
- Search bar visible in navbar with Cmd/Ctrl+K keyboard shortcut (auto-injected by plugin)
- Full-content indexing of all 423 pages across both `/docs/` and `/api/` instances
- Modal overlay with dimmed backdrop, type-ahead instant results, context snippets on every keystroke
- Click-to-navigate to both documentation and API endpoint pages
- Search result highlighting on target page (yellow marks on matched terms)
- Hashed index filenames (`hashed: true`) for CloudFront CDN cache compatibility
- Mobile-responsive modal across all target viewports (320px through 1440px)
- 1NCE CSS branding overrides (navy/teal palette, Barlow font, light-only mode enforcement on search overlay)

**Should have (add after validation — v1.4.x):**
- Separate search contexts for docs vs API via `searchContextByPaths` — reduces result noise across 423 mixed-type pages
- PostHog search event tracking (`posthog.capture`) — identify zero-result queries and content gaps
- Stop word filter removal (`removeDefaultStopWordFilter: true`) — preserves technical meaning of "NOT", "no", "is" in IoT/API content
- Redirect pages excluded from index via `ignoreFiles` — 80+ redirect stubs in this project would pollute results with duplicate entries

**Defer (v2+):**
- Algolia DocSearch: migration is straightforward (same navbar SearchBar slot); defer until budget/approval exists
- AI-powered "Ask AI" search: adds external API cost and latency; not justified for 423 pages
- Custom swizzled search UI: CSS customization via `--search-local-*` variables is sufficient without swizzling

### Architecture Approach

The integration is additive and non-destructive. The search theme plugs into Docusaurus's existing `SearchBar` component slot — no navbar files, sidebar configs, or existing plugins require modification. At build time, the plugin's `postBuild` hook scans all generated HTML in `build/`, extracts text, builds a lunr.js inverted index, and writes hashed JSON chunks to `build/`. At runtime, the browser fetches the relevant index chunk on first search interaction, queries it in a Web Worker, and returns ranked results. CloudFront serves the hashed JSON files as immutable static assets under the existing 1-year cache policy.

**Major components:**
1. **Search Theme** (`@easyops-cn/docusaurus-search-local`) — registered in `themes` array; provides SearchBar React component and `postBuild` index generator
2. **lunr.js Web Worker** — runs queries off the main thread to prevent UI jank; bundled with the theme; loaded lazily on first search interaction
3. **search-index-{hash}.json** — build artifact written to `build/`; served by CloudFront; content-hashed filename enables 1-year immutable CDN caching
4. **`docusaurus.config.ts` themes entry** — the single integration point; `docsRouteBasePath: ['/docs', '/api']` is the critical multi-instance configuration key
5. **`src/css/custom.css` overrides** — required for 1NCE branding and to prevent dark-mode bleed on the light-only site

**Components that do NOT change:**
- `src/theme/Navbar/Layout/index.tsx` — SearchBar renders via Docusaurus component slot, not the custom NavbarLayout
- `src/components/SecondaryNavbar/` — search goes in the primary navbar, not the tab bar
- GitHub Actions deploy workflow — `npm run build` already triggers `postBuild` hooks; no CI/CD changes needed
- CloudFront Function — search index JSON files are static assets, not SPA routes; no rewrite logic needed

### Critical Pitfalls

1. **API pages silently not indexed** — configure `docsRouteBasePath: ['/docs', '/api']` as an array (not the default single string `/docs`); after first build, grep the raw search-index JSON for known API titles ("Obtain Access Token", "Get SIM Details") before proceeding; this is a hard go/no-go gate for Phase 1

2. **Rspack (`experimental_faster`) incompatibility** — the plugin has zero reported usage with Rspack as of April 2026; run `npm run build` immediately after install and watch for build warnings; if search renders but shows no results, disable `experimental_faster` as the immediate fallback and file an upstream issue; do not discover this in production

3. **Oversized search index** — 125 API pages with multi-language code samples can push the uncompressed index above 5 MB; measure index file size after first build; verify CloudFront compression includes `application/json`; use `ignoreFiles` to exclude the 80+ redirect stub pages that would inflate the index with zero search value

4. **CloudFront serving stale/missing index post-deploy** — `aws s3 sync --delete` removes old hashed index files before invalidation fully propagates (1-5 min window); the existing smoke test should verify the search-index URL returns 200 after staging deploy; accept the short window or implement a two-phase deploy (upload new files, invalidate, then delete old)

5. **Dark mode bleed on light-only site** — the search plugin may respond to `prefers-color-scheme: dark` independently of Docusaurus's `colorMode` config; test with OS dark mode enabled on macOS and iOS; add CSS overrides forcing light backgrounds on search overlay components in `custom.css`

## Implications for Roadmap

Based on research, the implementation has a clear 3-phase dependency chain with a hard gate between Phase 1 and Phase 2.

### Phase 1: Plugin Installation and Core Configuration

**Rationale:** Everything else depends on the plugin being installed and both docs instances being indexed. The multi-instance configuration and Rspack compatibility check are the highest-risk steps and must be proven before any styling or optimization work begins. Phase 1 is gated by two acceptance criteria that must both pass before Phase 2 starts.

**Delivers:** Search bar in navbar, full index of all 423 pages (docs + API), Cmd+K keyboard shortcut, modal overlay with type-ahead results, click-to-navigate to both doc and API pages

**Addresses:** All P1 must-have features from FEATURES.md (core search functionality, multi-instance indexing, keyboard shortcut, modal, navigation)

**Avoids:** Pitfall 1 (API pages not indexed — acceptance criterion), Pitfall 3 (Rspack incompatibility — acceptance criterion)

**Acceptance criteria:**
- `npm run build` succeeds with `experimental_faster: true` and no search-related warnings
- Grep of `build/search-index-*.json` contains titles from known API endpoints ("Obtain Access Token", "Get SIM Details")
- Clicking an API search result navigates to the correct `/api/` page

### Phase 2: UI Integration, Styling, and Index Optimization

**Rationale:** Styling is blocked on Phase 1 (the plugin must be installed to test the UI). Index optimization is blocked on knowing the actual index size from a real build. Responsive layout testing at multiple viewports requires the actual search bar to exist.

**Delivers:** 1NCE-branded search overlay (navy/teal palette, Barlow font, light-only mode enforced), correct navbar layout at all viewports (320px–1440px), optimized index with redirect pages excluded, CloudFront JSON compression verified

**Addresses:** CSS branding (P1 feature), dark mode bleed prevention (Pitfall 6), navbar overflow at tablet widths (Pitfall 5), oversized index risk (Pitfall 2), redirect page pollution (80+ stubs)

**Implements:** `custom.css` search overlay overrides, `ignoreFiles` exclusions for redirect stub pages, `searchContextByPaths` config if mixed-result noise is confirmed by UX testing, CloudFront `application/json` compression verification

### Phase 3: Deployment Integration and Smoke Testing

**Rationale:** Deploy verification is only meaningful after the search UI is fully configured and styled. The CDN caching race condition (Pitfall 4) must be tested against actual S3/CloudFront staging — it is undetectable with `docusaurus serve` locally.

**Delivers:** Post-deploy smoke test verifying search-index URL returns 200 on staging CloudFront, confirmed CloudFront gzip/brotli compression for `application/json`, documented test procedure for search (build required; no dev-mode search) added to CONTRIBUTING

**Addresses:** Pitfall 4 (stale/missing index after deploy), CloudFront compression for JSON content type, developer onboarding clarity on `npm run build && npm run serve` requirement for search testing

### Phase Ordering Rationale

- Phase 1 is the dependency for all downstream work — you cannot style or optimize a search bar that does not exist and has not been verified to index both content types.
- The Rspack compatibility check is a Phase 1 gate. If Rspack must be disabled, that affects build time expectations and must be decided before Phase 2 begins work on CSS that may look different between Rspack and Webpack builds.
- Index optimization in Phase 2 must be measurement-driven, not speculative — only after seeing the actual index size can `ignoreFiles` exclusions be targeted correctly.
- Deployment smoke testing in Phase 3 must run against real CloudFront because the CDN caching race condition cannot be reproduced locally.

### Research Flags

Phases requiring attention during planning or execution:

- **Phase 1:** Rspack compatibility has no community precedent. Have a documented fallback plan ready before starting: if `npm run build` with `experimental_faster: true` produces a non-functional search, disable `experimental_faster`, confirm search works, file an upstream issue. Do not block the phase on Rspack — the fallback is valid.
- **Phase 2:** Index size for 125 OpenAPI-generated pages with multi-language code samples is unknown until the Phase 1 build runs. Do not pre-populate `ignoreFiles` patterns — measure first, then target the highest-weight content selectively.

Phases with standard, well-documented patterns (no additional research needed):
- **Phase 3:** Smoke testing and CloudFront invalidation follow the identical pattern as the existing deploy pipeline. The only new addition is verifying a JSON file URL and a response header — no novel infrastructure patterns.

### Gaps to Address

- **Rspack compatibility:** No community reports of `@easyops-cn/docusaurus-search-local` running with `@docusaurus/faster` exist as of April 2026. Must be tested empirically on first build. Mitigation is pre-planned (disable `experimental_faster`), but the interaction is unverified. Resolution: test in Phase 1, decide immediately.

- **Actual index size:** Estimated 1-5 MB compressed for 423 pages, but 125 OpenAPI-generated pages with multi-language code samples may push this higher. Cannot be known until a real build runs. Resolution: measure during Phase 1 build, optimize in Phase 2 if needed.

- **CloudFront JSON compression status:** The existing CloudFront config compresses HTML/CSS/JS. Whether it also includes `application/json` for the search index is unverified. Resolution: check `Content-Encoding` response header on `search-index-*.json` during Phase 3 staging deploy.

- **`polyfillNodeModules` plugin interaction:** The project has an existing `configureWebpack` plugin for `path-browserify` polyfill. Multiple `configureWebpack` returns may not merge correctly under Rspack. If Rspack compatibility fails, check this interaction as a secondary cause after ruling out the plugin itself.

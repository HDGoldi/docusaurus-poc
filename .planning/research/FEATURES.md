# Feature Landscape: AI & Search Readiness (v1.3)

**Domain:** LLM discoverability, AI agent integration, and search crawler optimization for developer documentation
**Researched:** 2026-04-03
**Site:** help.1nce.com (298 docs pages + 125 API endpoint pages, Docusaurus 3.9.2)

---

## Table Stakes

Features that AI tools, search engines, and developers now expect. Missing = invisible to a growing class of consumers.

| Feature | Why Expected | Complexity | Depends On |
|---------|--------------|------------|------------|
| **robots.txt** with sitemap reference | Every documentation site needs one. Without it, crawlers have no explicit policy and Google Search Console flags it as missing. Docusaurus does NOT auto-generate robots.txt -- must be placed manually in `static/`. | Low | Existing sitemap.xml (already generated by Docusaurus preset-classic) |
| **AI crawler user-agent directives** in robots.txt | GPTBot, ClaudeBot, PerplexityBot, and others check robots.txt before indexing. An explicit Allow policy signals intent. Without it, behavior varies by bot. | Low | robots.txt |
| **llms.txt** at site root | The llmstxt.org specification is the emerging standard for LLMs to discover documentation structure. Stripe, Supabase, Vercel AI SDK, and most major developer platforms serve one. Without it, LLMs must scrape and guess. | Medium | Content taxonomy decisions (already established: 11 doc sections + 6 API specs) |
| **llms-full.txt** companion file | The spec defines this as the expanded version containing all documentation content in a single file. Critical for LLMs that want to ingest the full corpus in one request rather than following individual links. | Medium | llms.txt structure, build-time content aggregation |
| **Sitemap reference in robots.txt** | `Sitemap: https://help.1nce.com/sitemap.xml` -- a single line, universally expected. Google's official documentation explicitly recommends it. | Low | Existing sitemap.xml |

## Differentiators

Features beyond baseline expectations. Not required, but signal a documentation site that treats AI consumers as first-class.

| Feature | Value Proposition | Complexity | Depends On |
|---------|-------------------|------------|------------|
| **SKILL.md for 1NCE API integration** | The `npx skills add` ecosystem (skills.sh) has 91K+ skills and is supported by Claude Code, Cursor, GitHub Copilot, and 15+ AI agents. A SKILL.md teaches AI coding agents HOW to use 1NCE APIs -- auth flows, common patterns, gotchas -- not just what endpoints exist. This is procedural knowledge that llms.txt (declarative/structural) does not cover. | Medium-High | Deep understanding of auth flows (Authorization API OAuth token), common integration patterns, error handling |
| **Product-first llms.txt organization** | Organizing llms.txt by product domain (1NCE Connect vs 1NCE OS) rather than filesystem path. Stripe does this -- sections like "Payments", "Billing", "Connect" rather than "docs/api/v1". For an IoT platform, sections like "SIM Management", "Connectivity Services", "Network Services" match how developers think. | Medium | Content taxonomy (already established in sidebar) |
| **Custom llms-*.txt files per product area** | Generate `llms-connect.txt` and `llms-os.txt` for focused context windows. An LLM working on SIM management should not have 1NCE OS documentation consuming its context budget. The docusaurus-plugin-llms supports `customLLMFiles` for exactly this. | Medium | docusaurus-plugin-llms configuration, content categorization |
| **Per-page .md endpoints** | The llms.txt spec recommends that individual HTML pages offer a `.md` version at the same URL (e.g., `help.1nce.com/docs/sim-management.md`). Lets LLMs fetch clean markdown for specific pages instead of parsing HTML. | Medium-High | Docusaurus plugin or custom build step to generate .md mirrors |
| **Structured "Optional" section in llms.txt** | The spec defines an `## Optional` H2 section for URLs that can be skipped when shorter context is needed. Use for troubleshooting guides, terms/abbreviations -- content that is supplementary. | Low | llms.txt structure |
| **Build-time link generation for llms.txt** | Rather than hand-maintaining 298+ page links, use docusaurus-plugin-llms to auto-generate link sections while keeping hand-curated H1/blockquote/section headers. Hybrid approach: curated structure, generated links. | Medium | docusaurus-plugin-llms or custom build script |
| **skills.sh registry listing** | Publishing the SKILL.md to a GitHub repo and listing on skills.sh makes 1NCE discoverable via `npx skills find 1nce`. | Low (after SKILL.md exists) | SKILL.md authored, GitHub repo |

## Anti-Features

Features to explicitly NOT build.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| **Block AI crawlers** | Some sites block GPTBot/ClaudeBot to prevent training data harvesting. For developer documentation, this is counterproductive -- you WANT LLMs to know about your APIs. 1NCE benefits from AI assistants answering "how do I manage SIMs with 1NCE?" | Explicitly Allow all AI crawlers. The content is public documentation, not proprietary. |
| **Fully automated llms.txt with no curation** | Dumping 423 pages in filesystem order produces noise. LLMs have context limits. A flat list with auto-generated descriptions is worse than a curated overview. | Hand-curate H1, blockquote, and section structure. Auto-generate only the link lists within those curated sections. |
| **SKILL.md as a full API reference** | SKILL.md should teach PATTERNS, not duplicate the API reference. "Here is every endpoint" belongs in llms.txt/OpenAPI specs. | Focus SKILL.md on auth flow, common multi-step workflows, error handling patterns, gotchas. Link to full API docs for reference. |
| **.well-known/skills/ directory with index.json** | The skills.sh ecosystem does NOT use .well-known/skills/. Skills are distributed via GitHub repos and installed with `npx skills add owner/repo` or `npx skills add https://url/SKILL.md`. The .well-known convention is not part of the current skills spec. | Serve SKILL.md from a GitHub repo or host at help.1nce.com/SKILL.md for direct URL installation. |
| **Dynamic llms.txt generation at runtime** | S3 + CloudFront serves static files. No server-side rendering available. | Generate at build time via plugin or script. Output goes to `build/llms.txt`. |
| **Crawl-delay directives** | Crawl-delay is non-standard (not in the original robots.txt RFC) and ignored by Googlebot. For a static CDN-served site, there is no server load concern. | Omit Crawl-delay entirely. |
| **Separate robots.txt per path** | robots.txt is per-domain, not per-path. Cannot have `/docs/robots.txt`. | Single robots.txt at site root via `static/robots.txt`. |
| **Overly granular Disallow rules** | Blocking specific paths like `/docs/internal/*` adds complexity and signals sensitive content. All content on help.1nce.com is public. | Simple `Allow: /` policy. If pages should not be crawled, do not publish them. |

## Feature Deep Dives

### llms.txt: Recommended Section Structure for 1NCE

Based on the llmstxt.org specification and real-world examples (Stripe uses product-first sections, Supabase uses minimal linking to expanded files):

**Required elements:**
- H1: `# 1NCE Developer Hub`
- Blockquote: 2-3 sentence summary -- what 1NCE is (IoT connectivity platform), what the docs cover, note about API authentication via OAuth

**Recommended H2 sections (product-first organization):**

| Section | Content | Maps To Site Content |
|---------|---------|----------------------|
| `## Getting Started` | Introduction, quickstart, platform overview | docs/documentation/introduction/ |
| `## SIM Management` | SIM lifecycle, activation, status management | docs/documentation/sim-cards/ + API: sim-management |
| `## Connectivity Services` | Data connectivity, SMS, device communication | docs/documentation/connectivity-services/ |
| `## Network Services` | Network configuration, APN settings | docs/documentation/network-services/ |
| `## Platform Services` | Data Streamer, SMS Forwarder, cloud integration | docs/documentation/platform-services/ |
| `## 1NCE Portal` | Portal UI guides, account management | docs/documentation/1nce-portal/ |
| `## 1NCE OS` | Software services, cloud integration, device management | docs/documentation/1nce-os/ |
| `## API Reference` | Links to all 6 API spec sections (Authorization, SIM Management, Order Management, Product Information, Support Management, 1NCE OS) | docs/api/ subdirectories |
| `## Blueprints & Examples` | Hardware integration tutorials, device examples | docs/documentation/blueprints-examples/ |
| `## Optional` | Troubleshooting, MCP server docs, terms/abbreviations | docs/documentation/troubleshooting/, mcp-server/ |

**Link format per spec:**
```markdown
- [SIM Activation Guide](https://help.1nce.com/docs/documentation/sim-cards/activation): How to activate and manage SIM cards for IoT devices
```

### SKILL.md: Recommended Structure for API-Heavy Documentation

Based on the skills.sh specification (verified via `npx skills init` template):

**Frontmatter (YAML):**
```yaml
---
name: 1nce-api-integration
description: Authenticate and interact with 1NCE IoT connectivity APIs -- SIM management, connectivity services, order management. Covers OAuth token flow, common multi-step patterns, and error handling.
---
```

**Body sections (procedural knowledge, NOT reference):**

| Section | Content | Rationale |
|---------|---------|-----------|
| `# 1NCE API Integration` | One-liner: what this skill teaches | Required by spec |
| `## When to use` | "Use when building IoT device management, SIM provisioning, or connectivity monitoring with 1NCE APIs" | Agents need trigger conditions to know when to activate this skill |
| `## Authentication` | Step-by-step OAuth flow: POST to /oauth/token with credentials, extract Bearer token, include in Authorization header for all subsequent calls | Auth is the #1 blocker for API integration. Without this, agents cannot make a single successful call. |
| `## API Base URLs` | Production endpoint URLs for each API group (Authorization, SIM Management, Order Management, Product Information, Support Management, 1NCE OS) | Agents need to know WHERE to call |
| `## Common Patterns` | Multi-step workflows: "To provision a new SIM: 1) Authenticate, 2) Create order, 3) Activate SIM, 4) Configure APN, 5) Verify connectivity" | Agents need to chain API calls correctly |
| `## Error Handling` | Common error codes (401 expired token, 403 insufficient permissions, 429 rate limit), what they mean in 1NCE context, retry strategies | Prevents agent loops on recoverable errors |
| `## Gotchas` | CORS limitations with browser-based calls (known issue), rate limits, SIM state machine constraints (cannot skip states), token expiry timing | Prevents common mistakes that waste agent cycles |
| `## Links` | Pointers to full API reference at help.1nce.com, OpenAPI spec URLs | Agents can deep-dive when the skill's procedural knowledge is insufficient |

### robots.txt: Recommended Content

```
# 1NCE Developer Hub - help.1nce.com
# Public developer documentation -- all crawlers welcome

User-agent: *
Allow: /

# Explicitly welcome AI crawlers for documentation indexing
User-agent: GPTBot
Allow: /

User-agent: ChatGPT-User
Allow: /

User-agent: ClaudeBot
Allow: /

User-agent: Claude-SearchBot
Allow: /

User-agent: Claude-User
Allow: /

User-agent: PerplexityBot
Allow: /

User-agent: Perplexity-User
Allow: /

User-agent: OAI-SearchBot
Allow: /

User-agent: Google-Extended
Allow: /

User-agent: Bravebot
Allow: /

# Sitemap
Sitemap: https://help.1nce.com/sitemap.xml
```

**Why explicit Allow for each AI agent:** While `User-agent: * / Allow: /` technically covers everything, explicitly listing AI crawlers: (1) signals clear intent to AI companies that this content should be indexed, (2) future-proofs against changes in default bot behavior, (3) serves as documentation of which bots exist. For a developer docs site, being maximally welcoming to AI is the correct posture.

```
robots.txt (standalone -- no dependencies)
   |
   +-- Sitemap reference (depends on: existing sitemap.xml -- already present)

llms.txt (depends on: content taxonomy decisions)
   |
   +-- llms-full.txt (depends on: llms.txt structure + build-time aggregation)
   |
   +-- Custom llms-connect.txt / llms-os.txt (depends on: plugin config)
   |
   +-- Per-page .md endpoints (depends on: plugin or custom build step)

SKILL.md (depends on: understanding of auth flows + common API patterns)
   |
   +-- skills.sh listing (depends on: SKILL.md + GitHub hosting)
```

**No circular dependencies.** All three top-level features (robots.txt, llms.txt, SKILL.md) are independent of each other and can be implemented in parallel.

## Implementation Approach: Plugin vs Custom

| Concern | docusaurus-plugin-llms (v0.3.0) | Custom Build Script |
|---------|--------------------------------|---------------------|
| llms.txt generation | Auto from docs content, configurable via `includeOrder` and `customLLMFiles` | Full control, hand-curated |
| llms-full.txt | Built-in, handles content cleaning and import removal | Must handle markdown extraction, HTML stripping manually |
| Custom per-product files | Supported via `customLLMFiles` config | Custom implementation needed |
| Maintenance | Auto-updates when content changes (new pages included automatically) | Requires manual updates or custom CI step |
| Content cleaning | Removes HTML, import statements, deduplicates headings | Must implement all cleaning logic |
| Batch processing | Built-in for large sites (default: 100 docs per batch) | Must handle memory for 423 pages |
| **Recommendation** | **Use this.** Install plugin, configure product-first sections. Hand-curate the H1/blockquote preamble. | Use only for robots.txt (static file) and SKILL.md (requires human authoring). |

**Plugin details:**
- Package: `docusaurus-plugin-llms` v0.3.0 (published 2026-02-08, MIT license)
- Author: Patrick Rachford (GitHub: rachfop/docusaurus-plugin-llms)
- Peer dependency: `@docusaurus/core ^3.0.0` (compatible with current 3.9.2)
- Generates: `llms.txt`, `llms-full.txt`, custom files -- all in `build/` output

**Example configuration for 1NCE:**
```javascript
['docusaurus-plugin-llms', {
  generateLLMsTxt: true,
  generateLLMsFullTxt: true,
  title: '1NCE Developer Hub',
  description: 'Documentation for the 1NCE IoT connectivity platform. Covers SIM management, connectivity services, network configuration, API references, and integration guides for IoT device connectivity.',
  includeOrder: [
    'documentation/introduction/**',
    'documentation/sim-cards/**',
    'documentation/connectivity-services/**',
    'documentation/network-services/**',
    'documentation/platform-services/**',
    'documentation/1nce-portal/**',
    'documentation/1nce-os/**',
    'api/**',
    'documentation/blueprints-examples/**',
    'documentation/troubleshooting/**',
  ],
  customLLMFiles: [
    {
      filename: 'llms-connect.txt',
      includePatterns: [
        'documentation/sim-cards/**',
        'documentation/connectivity-services/**',
        'documentation/network-services/**',
        'api/sim-management/**',
        'api/authorization/**',
        'api/order-management/**',
        'api/product-information/**',
        'api/support-management/**',
      ],
      fullContent: true,
      title: '1NCE Connect Documentation',
    },
    {
      filename: 'llms-os.txt',
      includePatterns: ['documentation/1nce-os/**', 'api/1nce-os/**'],
      fullContent: true,
      title: '1NCE OS Documentation',
    },
  ],
}]
```

## MVP Recommendation

### Phase 1: Crawl Foundation (Low effort, immediate impact)
1. **`static/robots.txt`** -- Allow all crawlers, reference sitemap. A static text file placed in `static/`. **30 minutes.**
2. **`docusaurus-plugin-llms` installation and basic config** -- Install plugin, configure `includeOrder` matching content taxonomy. Generates llms.txt + llms-full.txt at build time. **2-4 hours.**

### Phase 2: Curated AI Presence (Medium effort, high differentiation)
3. **Hand-curate llms.txt preamble** -- Override auto-generated H1/blockquote with product-first language about 1NCE IoT platform, authentication note. **1 hour.**
4. **Custom llms-connect.txt and llms-os.txt** -- Configure `customLLMFiles` in plugin for focused context files. **1-2 hours.**

### Phase 3: Agent Integration (Higher effort, strong differentiation)
5. **SKILL.md for API integration** -- Author procedural knowledge: auth flows, common patterns, gotchas. This is content authoring, not code. Requires testing with actual API workflows. **4-8 hours.**

### Defer
- **Per-page .md endpoints**: High complexity, unclear incremental value beyond llms-full.txt. Revisit after measuring LLM traffic patterns.
- **skills.sh registry listing**: Trivial once SKILL.md exists, but requires deciding on hosting (same repo vs separate). Do after SKILL.md is validated with at least one AI agent.
- **.well-known/skills/ directory**: Not part of the actual skills.sh specification. Do not build.

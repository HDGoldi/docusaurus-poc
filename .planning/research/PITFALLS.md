# Pitfalls Research

**Domain:** Adding client-side full-text search to Docusaurus 3.9.2 with Rspack and docusaurus-openapi-docs
**Researched:** 2026-04-05
**Confidence:** MEDIUM-HIGH

## Critical Pitfalls

### Pitfall 1: API Endpoint Pages Not Appearing in Search Results

**What goes wrong:**
The site uses two separate docs plugin instances: `preset-classic` (routeBasePath `/docs`, 298 pages) and a standalone `@docusaurus/plugin-content-docs` with `id: 'api'` (routeBasePath `/api`, 125 pages). After installing `@easyops-cn/docusaurus-search-local`, the 125 API endpoint pages generated by `docusaurus-openapi-docs` silently do not appear in search results. Developers search for "Get SIM Details" or "Obtain Access Token" and get zero hits.

**Why it happens:**
The search plugin indexes all docs plugin instances by default when `indexDocs: true`. However, the real problem is twofold: (1) OpenAPI-generated pages have a different content structure (schema tables, code samples in 5+ languages, parameter definitions) that the text indexer may not parse cleanly, and (2) older versions of `docusaurus-openapi-docs` had bugs preventing docs from being generated outside the main `/docs` directory, which broke `searchContextByPaths` filtering. GitHub issue #550 on the search plugin repo confirmed this exact scenario was reported and required a plugin upgrade to resolve (Dec 2025).

**How to avoid:**
1. After first build with the search plugin, inspect the generated search index file. Search the raw JSON for known API endpoint titles. If absent, the API pages are not being indexed.
2. Configure `searchContextByPaths` to explicitly separate the two content areas:
   ```ts
   searchContextByPaths: [
     { label: 'Documentation', path: '/docs' },
     { label: 'API Explorer', path: '/api' },
   ]
   ```
3. Do NOT set `indexDocs: false` thinking it only affects the main docs instance -- it disables indexing across all docs plugin instances.
4. The project is already on `docusaurus-plugin-openapi-docs` v4.7.1, which should handle multi-directory generation correctly. Verify after installation.

**Warning signs:**
- Searching for any API endpoint name returns zero results.
- The search index file is suspiciously small (under 1 MB for 423 pages strongly suggests missing content).
- Search results only ever show `/docs/` paths, never `/api/` paths.

**Phase to address:**
Phase 1 (Plugin Installation and Configuration). This is a go/no-go acceptance criterion -- do not proceed until API pages appear in search.

### Pitfall 2: Oversized Search Index Degrading Page Load and Runtime Performance

**What goes wrong:**
With 423 pages (298 docs + 125 API), the search index file can easily reach 5-30 MB uncompressed. API pages generated from OpenAPI specs are disproportionately heavy -- each endpoint page contains request/response JSON schemas, code samples in multiple languages (curl, Python, JS, Java, Go), and detailed parameter tables. The index is fetched client-side, parsed in the browser, and held in memory. On slow connections or mobile devices, search becomes unusable. GitHub issue #549 documents a site with a 30 MB uncompressed index where "search becomes very slow until refresh" -- the maintainer confirmed the root cause was the extremely large index without compression enabled.

**Why it happens:**
`docusaurus-search-local` builds a lunr.js index from the full text content of every page, serialized as a single JSON file. OpenAPI-generated pages contain extensive repetitive content -- JSON schema definitions that repeat across endpoints, code snippets in 5+ languages per endpoint, verbose parameter descriptions. A 125-endpoint API suite can easily contribute 60-70% of the total index weight despite being only 30% of the pages.

**How to avoid:**
1. Measure immediately after first build: check the size of the search index file in `build/`. If over 5 MB uncompressed, take action.
2. Enable gzip/brotli compression on CloudFront for `application/json` content type. The project already uses CloudFront -- verify the compress settings include JSON. A 10 MB JSON typically compresses to 1-2 MB with gzip, 0.5-1 MB with brotli.
3. Use the `ignoreFiles` option to exclude content that should not be searchable (e.g., redirect stub pages from `plugin-client-redirects`).
4. Tune `searchResultLimits` (default 8) and `searchResultContextMaxLength` (default 50) to control how much context is stored per match.
5. Use `hashed: true` (the default) so the index filename includes a content hash for proper CDN cache busting.
6. If the index exceeds 10 MB even after compression, consider whether all code samples need indexing. The plugin does not natively support excluding specific page sections, but custom `ignoreFiles` patterns can exclude the most verbose endpoint pages.

**Warning signs:**
- Search index file exceeds 5 MB uncompressed in `build/`.
- Lighthouse reports a large network payload warning.
- Users on mobile report search spinning or freezing.
- Browser DevTools shows a multi-second JSON parse for the search index.

**Phase to address:**
Phase 2 (Index Optimization). Measure after Phase 1 install, optimize in Phase 2 if needed. Do not prematurely optimize -- the index may be acceptable.

### Pitfall 3: Rspack Bundler Incompatibility with Search Plugin

**What goes wrong:**
The project uses `experimental_faster: true` (Rspack bundler via `@docusaurus/faster`). The search plugin uses `configureWebpack` in its plugin lifecycle to inject client-side search components and may use Webpack-specific loader configurations. While Rspack implements most of the Webpack 5 API, edge cases in loader configuration, module resolution, or plugin hooks can cause silent build failures or runtime errors where the search bar renders but produces no results.

**Why it happens:**
Rspack is not 100% Webpack-compatible. The `docusaurus-search-local` plugin was developed and tested against Webpack. The project already has a `polyfillNodeModules` plugin using `configureWebpack` for `path-browserify` -- interactions between multiple `configureWebpack` plugins under Rspack can produce unexpected merge behavior. Critically, a search of the plugin's GitHub issue tracker (April 2026) returned ZERO results for "rspack" or "faster". This means either it works seamlessly (best case) or nobody has tried this specific combination and reported issues (unknown risk).

**How to avoid:**
1. After installing the search plugin, immediately run `npm run build` and watch for Rspack warnings or errors in the build output.
2. Test search in the production build output (`npm run build && npm run serve`), not just dev mode. Rspack and Webpack can behave differently between dev and production builds.
3. If the build fails or search does not work at runtime, test with `experimental_faster` temporarily disabled. If that fixes it, you have confirmed an Rspack incompatibility -- file an upstream issue and decide whether to keep Rspack disabled or find a workaround.
4. Verify the search plugin version is 0.52.2+ (the version that fixed Docusaurus 3.9+ compatibility per issue #542).

**Warning signs:**
- Build warnings about unsupported loader options or resolve configurations.
- Search bar renders but shows "Loading..." indefinitely.
- Search works in `docusaurus start` (Webpack dev server) but fails in `docusaurus build && docusaurus serve` (Rspack production build).
- Console errors related to missing modules or worker initialization.

**Phase to address:**
Phase 1 (Plugin Installation). This is an immediate go/no-go check. If Rspack is incompatible, the options are: (a) fix the plugin, (b) temporarily disable `experimental_faster`, or (c) choose a different search solution.

### Pitfall 4: CloudFront Serving Stale or Missing Search Index After Deploy

**What goes wrong:**
After a new deployment, users get broken search because CloudFront serves a cached version of the HTML that references an old search index filename, but the old index file has been deleted from S3 by `aws s3 sync --delete`. The result is a 404 on the search index file, and search silently fails.

**Why it happens:**
With `hashed: true` (default), the search index filename includes a content hash (e.g., `search-index-abc123.json`). When content changes, the hash changes, producing a new filename. The HTML referencing the index is cached for 10 minutes (`max-age=600, must-revalidate`). During the deploy window: (1) new files are synced to S3 with `--delete` flag, removing the old index file, (2) CloudFront invalidation runs on `/*`, but edge propagation takes 1-5 minutes, (3) users hitting edge caches that haven't received the invalidation yet get old HTML that references the now-deleted old index file -- resulting in a 404.

**How to avoid:**
1. The project already runs `aws cloudfront create-invalidation --paths "/*"` in the deploy workflow. This is necessary but not instantaneous.
2. Consider a deploy strategy where new files are uploaded first, then old files are cleaned up after invalidation completes. This is more complex but eliminates the race condition.
3. Alternatively, accept the 1-5 minute window of potential breakage. For a documentation site, this is usually acceptable.
4. Add a post-deploy smoke test that waits 60 seconds, then verifies the search index URL returns 200. Add the search index URL pattern to the existing smoke test script.

**Warning signs:**
- Search fails immediately after deploy but works after a hard refresh.
- Browser console shows 404 for a `search-index-*.json` file.
- Works on direct S3 access but not through CloudFront.

**Phase to address:**
Phase 3 (Deployment Integration). Add search index verification to the existing smoke test.

### Pitfall 5: Search Bar Breaking Navbar Layout on Tablet and Mobile

**What goes wrong:**
The current navbar has a logo on the left and three external links ("1NCE Home", "1NCE Shop", "1NCE Portal") on the right. The search bar needs to be added "next to Terms and Abbreviations" on the right side. Adding the search plugin injects a `SearchBar` component into the navbar's right section. With three existing right-side items plus a search input, the navbar runs out of horizontal space at tablet widths (768-1024px), causing items to wrap to a second line or overflow off-screen. On mobile, the search bar may conflict with the hamburger menu.

**Why it happens:**
`docusaurus-search-local` injects the search bar into the navbar automatically. The default search input has a minimum width that, combined with the existing three external links, exceeds the available space at medium viewport widths. The dark navy navbar (`--ifm-navbar-background-color: #194a7d`) makes overflow harder to spot visually because the overflow clips against a dark background.

**How to avoid:**
1. Test at specific breakpoints after installation: 320px (mobile), 375px (iPhone SE), 768px (iPad portrait), 1024px (iPad landscape), 1440px (desktop).
2. Constrain the search input width via CSS:
   ```css
   .navbar [class*="searchBar"] {
     max-width: 200px;
   }
   @media (max-width: 996px) {
     .navbar [class*="searchBar"] {
       max-width: 160px;
     }
   }
   ```
3. Consider whether all three external links need to stay in the navbar at mobile widths. Moving them to the hamburger menu on mobile frees space for search.
4. Test the mobile search overlay: tapping search on mobile should open a full-screen overlay, not try to expand inline in the cramped navbar.

**Warning signs:**
- Navbar items wrap to two lines on iPad.
- External links disappear or are clipped on smaller screens.
- Search bar is invisible on mobile (buried in hamburger menu without proper integration).

**Phase to address:**
Phase 2 (UI Integration and Styling). Must test across all target viewports.

### Pitfall 6: Search Overlay Ignoring Light-Only Mode

**What goes wrong:**
The site enforces light-only mode (`disableSwitch: true`, `respectPrefersColorScheme: false`). The search plugin ships with both light and dark theme styles. If the plugin independently detects `prefers-color-scheme: dark` from the user's OS, the search overlay renders with a dark background and light text on a site that is otherwise entirely light-themed. The visual mismatch is jarring and looks like a bug.

**Why it happens:**
The search plugin's CSS may include `[data-theme='dark']` selectors or `@media (prefers-color-scheme: dark)` queries that activate based on OS settings, independent of Docusaurus's `colorMode` configuration. The existing `custom.css` has `[data-theme='dark']` overrides (lines 42-49) that only affect Infima variables, not the search plugin's styles.

**How to avoid:**
1. After installation, set your OS to dark mode and test the search overlay. If it goes dark, add CSS overrides.
2. Target the search plugin's components in `custom.css` to force light mode:
   ```css
   /* Force light mode on search overlay regardless of OS preference */
   [class*="searchModal"],
   [class*="searchResult"] {
     background-color: #fff !important;
     color: var(--ifm-font-color-base) !important;
   }
   ```
3. Check if the plugin has a `forceTheme` or `colorMode` configuration option.

**Warning signs:**
- Search overlay appears with dark/black background when user's OS is in dark mode.
- Search result text is unreadable (low contrast).
- Only reproducible on machines with dark OS theme.

**Phase to address:**
Phase 2 (UI Integration and Styling).

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Skip index size measurement after first build | Faster initial setup | Discover performance problems in production when real users complain on mobile | Never -- always measure after first build |
| Index all content without filtering | Complete search coverage | Bloated index with repetitive code samples and schema definitions from 125 API pages | MVP only -- measure and optimize before production launch |
| Use default search UI without custom CSS | Zero styling effort | Dark mode bleed-through, navbar overflow on tablet, brand mismatch | Never for this project -- light-only mode and branded navbar require customization from day one |
| Skip multi-instance docs verification | Assume "indexDocs: true indexes everything" | 125 API pages invisible in search, discovered after launch | Never -- verify in Phase 1 as acceptance criterion |
| Hardcode search plugin version without pinning | Simpler package.json | Patch update introduces breaking change (like the 3.9.1 compat break in v0.52.1) | Never -- pin exact version after verifying it works |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| search-local + openapi-docs (multi-instance) | Assuming API pages are automatically indexed | Build, then search the raw index file for known API endpoint titles like "Obtain Access Token" |
| search-local + plugin-client-redirects | Redirect stub pages get indexed, polluting search with duplicates | Use `ignoreFiles` to exclude redirect paths (the project has 80+ redirects that generate HTML stubs) |
| search-local + CloudFront | Not compressing JSON files at CDN edge | Verify CloudFront behavior compresses `application/json`; the existing config may only compress HTML/CSS/JS |
| search-local + Rspack (`experimental_faster`) | Zero prior art in the community | Test build immediately after install; have fallback plan to disable Rspack if search breaks |
| search-local + llms-txt-plugin | Both use `postBuild` lifecycle; undefined execution order | Verify both produce correct output; order should not matter but confirm |
| search-local + polyfillNodeModules plugin | Multiple `configureWebpack` returns may not merge correctly under Rspack | If search fails, check whether the path-browserify polyfill is interfering; test removing it temporarily |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Full search index loaded before first interaction | Increased page weight and Time to Interactive for all users, even those who never search | Verify the plugin lazily loads the index on first search interaction (not on page load) | Index > 3 MB on mobile 3G connections |
| OpenAPI code samples inflating index 3-5x | Index dominated by code snippets; search results show irrelevant code matches instead of prose | Measure API-page contribution to index; use `ignoreFiles` if code samples dominate | 125 API pages with multi-language code samples |
| No search result context highlighting on target page | Users click a result, land on a long API page, cannot find the matched text | Verify the plugin's highlight-on-navigate feature is enabled and works with OpenAPI page structure | All page counts -- UX problem, not scale problem |
| Browser memory pressure from large parsed index | Tab becomes sluggish after extended use; search slows down over time until page refresh | Enable compression; keep index under 5 MB uncompressed | Index > 10 MB; mobile Safari (aggressive memory limits) |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Search index exposing draft/unpublished content | Internal draft pages appear in search on production site | Build with `NODE_ENV=production`; verify `search-index.json` only contains published pages |
| Search index containing internal file paths | Filesystem paths leak in index metadata, revealing server directory structure | Inspect raw search index JSON for any absolute file paths before first deploy |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Search results do not distinguish docs vs API pages | Users cannot tell if a result is documentation or an API endpoint; they click and end up in an unexpected context | Use `searchContextByPaths` to add visible labels ("Documentation" / "API Explorer") to search results |
| No "no results found" feedback | Users type a query, see empty space, assume search is broken | Verify the plugin shows a clear "No results found" message; customize the text if generic |
| Search overlay blocks navigation on mobile | Users cannot close search or navigate away on small screens; requires hard back-button press | Test overlay dismiss behavior (ESC key, tap-outside, X button) on mobile Safari and Chrome |
| Anchor scroll offset wrong after navigating from search | Users click a heading-level search result but the content is hidden behind the fixed 60px navbar | Verify `scroll-margin-top` is set on heading elements to account for navbar height |
| Keyboard shortcut conflicts | Cmd+K / Ctrl+K may conflict with browser shortcuts or other site features | Test the shortcut works and does not conflict; verify it is documented in the search bar hint |

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| API pages not indexed | LOW | Add `searchContextByPaths` config or adjust `docsPluginIdForPreferredVersion`, rebuild, redeploy |
| Index too large (>10 MB) | MEDIUM | Add `ignoreFiles` patterns for verbose API pages, tune `searchResultContextMaxLength`, rebuild. If still too large, consider Pagefind as alternative. |
| Rspack incompatibility | MEDIUM | Disable `experimental_faster` as immediate fix (build time increases but search works); file upstream issue; re-enable when patched |
| CloudFront serving stale/missing index | LOW | Run `aws cloudfront create-invalidation --paths "/*"`; wait 5 minutes; verify. Ensure `hashed: true` is set. |
| Navbar overflow on tablet | LOW | Add CSS `max-width` on search bar; adjust breakpoint behavior in `custom.css` |
| Dark mode bleed on search overlay | LOW | Add CSS overrides forcing light mode on search overlay components in `custom.css` |
| Redirect pages polluting search | LOW | Add redirect paths to `ignoreFiles` array, rebuild |

# Pitfalls Research

**Domain:** Docusaurus navigation restructuring, dark mode removal, and branding changes for v1.2
**Researched:** 2026-04-02
**Confidence:** HIGH (based on direct codebase analysis and Docusaurus architecture knowledge)

## Critical Pitfalls

### Pitfall 1: Broken URLs after merging docs plugin instances

**What goes wrong:**
The current site uses 4 separate `@docusaurus/plugin-content-docs` instances: `default` (docs/documentation at /docs), `platform` (docs/platform at /platform), `blueprints` (docs/blueprints at /blueprints), and `terms` (docs/terms at /terms). Merging platform, blueprints, and terms into the default docs instance changes every URL path for those sections. Pages currently at `/platform/1nce-os/...` would move to `/docs/1nce-os/...`. Every cross-reference, external bookmark, search engine index entry, and internal link in MDX files breaks silently.

**Why it happens:**
Docusaurus routes are derived from `routeBasePath` + the file path relative to the plugin's `path` directory. Changing which plugin owns a file changes its URL. Developers focus on getting the sidebar right and forget that URLs are a contract with the outside world.

**How to avoid:**
1. Map out every current URL before making changes (run `docusaurus build` and list all generated paths).
2. After merging, verify every page has the same or predictable new URL.
3. Add `@docusaurus/plugin-client-redirects` to create redirects from old paths (`/platform/*`, `/blueprints/*`, `/terms/*`) to new paths under `/docs/*`.
4. Configure the CloudFront Function to handle legacy redirects for paths that changed.
5. Run `docusaurus build` with `onBrokenLinks: 'throw'` to catch broken internal cross-references immediately.

**Warning signs:**
- Build warnings about broken links (currently set to `warn` -- must switch to `throw` during merge verification)
- 404 errors in smoke tests after deploy
- Analytics showing sudden traffic drop on previously popular pages

**Phase to address:**
Navigation restructuring phase -- this is the single most important thing to get right. Must be step 1, before any other changes.

### Pitfall 2: Sidebar ID conflicts and the root sidebars.ts confusion

**What goes wrong:**
The project has TWO competing sidebar sources. The preset's docs plugin uses `./sidebars/documentation.ts` (which is `autogenerated`), but there is also a root-level `sidebars.ts` that contains a massive manually curated `docsSidebar` with 700 lines of content including categories for "1NCE OS", "Blueprints & Examples", "API Reference", etc. This root file appears to be unused by the current config (no plugin references it). During the merge, a developer may edit the wrong sidebar file, or try to use both, causing either missing sidebar content or build failures from doc ID mismatches.

**Why it happens:**
The root `sidebars.ts` was likely the original sidebar definition from early development. When the project split content into multiple plugin instances, each got its own sidebar file in `sidebars/`, but the root file was never deleted. It now serves as a confusing red herring.

**How to avoid:**
1. Determine which sidebar file is active for each plugin (check `sidebarPath` in config -- the default docs plugin uses `./sidebars/documentation.ts`).
2. Delete the root `sidebars.ts` file -- it is unused and misleading. Or repurpose it as the new unified sidebar.
3. After merging plugins, build one consolidated sidebar. Do NOT use `autogenerated` for the main Documentation sidebar -- it needs manual curation to match the original hub's category structure.
4. Name the sidebar file clearly (e.g., `sidebars/main.ts`) to avoid future confusion.

**Warning signs:**
- Pages rendering with an empty or wrong sidebar
- Build errors about unknown sidebar IDs in navbar items
- Pages visible in sidebar but leading to 404s (sidebar references doc IDs that do not exist at the expected path)

**Phase to address:**
Navigation restructuring phase -- must be resolved simultaneously with the plugin merge.

### Pitfall 3: Navbar items referencing deleted plugin IDs crash the build

**What goes wrong:**
The current navbar has items like `{ type: 'docSidebar', sidebarId: 'platformSidebar', docsPluginId: 'platform' }`. After removing the `platform` plugin instance, this navbar item causes a build failure because Docusaurus cannot resolve `docsPluginId: 'platform'`. This affects 3 navbar items: 1NCE Platform, Blueprints & Examples, and Terms & Abbreviations.

**Why it happens:**
Navbar configuration and plugin configuration are in different sections of `docusaurus.config.ts`. Developers update one and forget the other. Docusaurus does not validate navbar items against available plugins until build time.

**How to avoid:**
1. Before removing any plugin, search the entire `docusaurus.config.ts` for ALL references to that plugin's ID (`'platform'`, `'blueprints'`, `'terms'`).
2. Remove the navbar items that pointed to deleted plugins. After merging, the Documentation tab should show everything in one unified sidebar.
3. The new navbar should have: Documentation (merged sidebar), API Explorer (unchanged), external links (new).
4. Make config changes and plugin removal in the same commit to avoid broken intermediate states.

**Warning signs:**
- Build failure: "Plugin with id 'platform' not found"
- Navbar items that silently disappear (Docusaurus suppresses some config errors in dev mode but fails in production build)

**Phase to address:**
Navigation restructuring phase -- must be done atomically with the plugin merge.

### Pitfall 4: Dark mode persists despite hiding the toggle

**What goes wrong:**
After setting `colorMode: { disableSwitch: true }`, the toggle disappears but the site STILL switches to dark mode for users with OS dark mode preference. The current config has `respectPrefersColorScheme: true`, which means the media query `prefers-color-scheme: dark` still triggers dark mode. The toggle is hidden but the behavior persists.

**Why it happens:**
Developers set `disableSwitch: true` and assume dark mode is gone. But `respectPrefersColorScheme` is an independent setting. With both `disableSwitch: true` and `respectPrefersColorScheme: true`, users with OS dark mode get forced into dark mode with NO way to switch back (because the toggle is hidden). This is the worst possible state.

**How to avoid:**
1. Set all three properties explicitly:
   ```typescript
   colorMode: {
     defaultMode: 'light',
     disableSwitch: true,
     respectPrefersColorScheme: false,
   }
   ```
2. Remove or comment out all `[data-theme='dark']` blocks in `custom.css` (lines 42-57, 64-67, 98-101).
3. Remove `darkTheme: prismThemes.dracula` from prism config (or leave it -- harmless dead code).
4. Test with OS dark mode enabled to verify the site stays light.

**Warning signs:**
- Flash of dark theme on page load for users with OS dark mode
- Users stuck in unreadable dark mode with no toggle to escape
- Code blocks using Dracula theme while rest of page is light (jarring contrast)

**Phase to address:**
Theme/branding phase -- should be a dedicated step, not mixed with navigation changes.

### Pitfall 5: OpenAPI docs plugin accidentally broken during merge

**What goes wrong:**
The `docusaurus-plugin-openapi-docs` is configured with `docsPluginId: 'api'`, linking it to the `api` docs plugin instance. This plugin instance is NOT being merged (it remains separate for the API Explorer tab). During the merge of other plugins, developers may accidentally modify or remove the `api` plugin, or change its ID, breaking the entire OpenAPI docs generation pipeline and 125 API endpoint pages.

**Why it happens:**
When removing 3 of 4 additional plugin instances, it is easy to accidentally modify the wrong one. The `api` plugin and the `openapi` plugin are tightly coupled -- breaking one breaks both. A fast "clean up all extra plugins" approach catches the `api` plugin in the crossfire.

**How to avoid:**
1. Explicitly mark the `api` plugin as "do not touch" during the merge.
2. Only remove the `platform`, `blueprints`, and `terms` plugin declarations.
3. Keep the `api` plugin, its navbar item, and its sidebar file exactly as they are.
4. After merging, run `npm run docusaurus gen-api-docs all` to verify OpenAPI docs still generate.
5. Run a build and verify the API Explorer tab renders correctly.

**Warning signs:**
- `gen-api-docs` command fails
- API Explorer pages show empty content or missing sidebars
- Build errors referencing `docItemComponent: '@theme/ApiItem'`

**Phase to address:**
Navigation restructuring phase -- include API Explorer verification in the merge checklist.

### Pitfall 6: Moving files without preserving doc IDs

**What goes wrong:**
When moving files from `docs/platform/`, `docs/blueprints/`, and `docs/terms/` into the unified docs directory, the doc IDs change because Docusaurus derives IDs from the file path relative to the plugin's `path` root. The sidebar definitions, any cross-links between docs, and any `doc` type navbar items referencing those IDs all break.

**Why it happens:**
Docusaurus doc IDs default to the file path minus extension, relative to the docs root. Moving `docs/platform/1nce-os/foo.md` to `docs/documentation/1nce-os/foo.md` could preserve the ID IF the relative path within the new root stays the same. But if you reorganize folders during the move (e.g., nesting platform content under a new `platform/` subdirectory within `docs/documentation/`), IDs change and every reference breaks.

**How to avoid:**
1. Before moving files, audit frontmatter for explicit `id` and `slug` fields.
2. Plan the target directory structure FIRST, then verify what IDs will result.
3. If IDs must change, update all sidebar references in one pass.
4. After moving, build immediately and fix any "doc not found" errors.
5. Consider adding explicit `slug` frontmatter to preserve old URLs while files live at new paths.

**Warning signs:**
- Build errors: "Document with id 'X' not found"
- Sidebar showing empty categories
- Pages accessible via direct URL but missing from sidebar navigation

**Phase to address:**
Navigation restructuring phase -- file moves must happen in a single atomic step with sidebar updates.

### Pitfall 7: External navbar links misconfigured as internal routes

**What goes wrong:**
Adding external links (1NCE Home, Shop, Portal) requires using `href` (external) instead of `to` (internal). Using `to` makes Docusaurus try to route them as client-side navigation, resulting in a Docusaurus 404 page. Additionally, external links in Docusaurus navbar do not automatically open in a new tab.

**Why it happens:**
Docusaurus navbar items support both `to` (internal) and `href` (external), and the distinction is easy to confuse. The documentation is not prominent about this difference.

**How to avoid:**
1. Use `href` (not `to`) for all external URLs.
2. Add external links as plain navbar items with the right properties:
   ```typescript
   {
     href: 'https://1nce.com',
     label: '1NCE Home',
     position: 'right',
   }
   ```
3. Docusaurus automatically adds `target="_blank"` and `rel="noopener noreferrer"` for `href` items pointing to external URLs. Do not fight this behavior.
4. Position external links on the right side (`position: 'right'`) to visually separate from internal navigation -- this is a standard UX convention.

**Warning signs:**
- Clicking a navbar link shows a Docusaurus 404 page instead of navigating externally
- Build errors about routes not matching any page

**Phase to address:**
Navbar enhancement phase -- straightforward but must be tested.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Leaving `[data-theme='dark']` CSS after disabling dark mode | No immediate effort | Dead CSS shipped to users, confusion for future contributors who think dark mode is supported | Acceptable as temporary step; clean up within same milestone |
| Using `autogenerated` sidebars for the merged Documentation section | Fast, no manual sidebar maintenance | No control over ordering, labels, or grouping; sidebar order determined by filesystem not editorial intent | Never -- the main sidebar needs manual curation to match original hub structure |
| Skipping redirect configuration for old URLs | Faster initial merge | SEO damage, broken bookmarks from existing users, broken links from external sites | Never -- redirects are cheap and critical |
| Keeping the root `sidebars.ts` alongside `sidebars/documentation.ts` | Avoid deleting code | Two competing sidebar definitions create ongoing confusion | Never -- delete the unused one during merge |
| Hardcoding new logo/favicon paths without cache busting | Simpler deploy | Users see old assets for hours/days due to CDN and browser caching | Acceptable if you issue CloudFront invalidation for `/img/*` and `/favicon.ico` during deploy |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| docusaurus-plugin-openapi-docs | Accidentally removing or modifying the `api` plugin instance during merge of other plugins | Leave the `api` plugin completely untouched; it is independent of the documentation restructuring |
| @docusaurus/plugin-client-redirects | Adding hundreds of individual redirect entries | Use the `createRedirects` function to programmatically generate redirects based on path patterns (e.g., any path starting with `/platform/` redirects to `/docs/` equivalent) |
| CloudFront Function | Not testing that the SPA routing function handles new and old paths correctly | Verify the CloudFront Function still rewrites paths correctly after the merge; old paths should either redirect or fall through to the redirect plugin |
| GitHub Actions smoke tests | Smoke test URLs hardcoded to old paths like `/platform/...` | Update smoke test expectations to use new paths AND add tests verifying old paths return proper redirects |
| Analytics (GTM/PostHog/SimpleAnalytics) | Path changes break analytics dashboards and historical data continuity | Document the path migration; note that `/platform/X` data maps to `/docs/X` going forward |
| Favicon / static assets | Deploying new favicon but CDN serves cached old one for days | Issue `CloudFront invalidation` for `/favicon.ico`, `/img/*` after deploying new branding assets |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Merged sidebar file becomes enormous | Slow dev server hot reload, large client-side JS bundle for sidebar metadata | Use collapsible categories aggressively; keep sidebar depth to 3-4 levels max | At 500+ sidebar items without nesting |
| Redirect plugin with too many individual entries | Increased build time, larger build output with redirect HTML files | Use `createRedirects` function with path patterns instead of individual entries | At 200+ explicit redirect entries |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| External navbar links without `rel="noopener noreferrer"` | Tab-napping attack vector | Docusaurus adds this automatically for `href` items pointing to external domains -- verify in rendered HTML but do not need to add manually |
| Serving favicon/logo from external CDN | Resource tampering if CDN compromised | Serve from local `static/img/` directory, not external URLs |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Merging all sections into one massive flat sidebar | Users cannot find content in an overwhelming wall of links | Use clear top-level categories with collapse-by-default for deeper sections; match the original hub's grouping |
| Removing dark mode without addressing the underlying readability issue | Users who preferred dark mode lose the option entirely | Acceptable -- PROJECT.md states dark mode is "not readable." Clean removal is correct. If dark mode is ever re-added, fix the color tokens first. |
| External links indistinguishable from internal navigation | Users confused when new tab opens | Position external links on navbar right side; Docusaurus adds an external-link icon by default for `href` items |
| Changing favicon without CloudFront invalidation | Users see stale favicon for hours/days | Issue targeted invalidation for favicon and logo paths after deploy |
| Sidebar categories that do not match original hub naming | Returning ReadMe.com users cannot find familiar sections | Use exact same category names as the original hub: "1NCE Portal", "SIM Cards", "Connectivity Services", "Platform Services", "1NCE OS", "Blueprints & Examples" etc. |

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Broken URLs after merge (no redirects) | MEDIUM | Add `@docusaurus/plugin-client-redirects` after the fact; issue CloudFront invalidation; submit updated sitemap to search engines. SEO recovery takes 2-4 weeks. |
| Dark mode still activating for OS dark mode users | LOW | Fix the `colorMode` config to include `respectPrefersColorScheme: false`, clean CSS, redeploy. No lasting damage. |
| OpenAPI docs broken by accidental api plugin change | MEDIUM | Restore the `api` plugin config from git history; re-run `gen-api-docs all`; rebuild. May need to regenerate all 125 API pages. |
| Sidebar shows wrong content or structure | LOW | Fix sidebar config and redeploy. No data loss, no lasting damage. |
| Navbar links broken or pointing to wrong places | LOW | Fix config and redeploy. Trivial fix. |
| Stale favicon/logo after deploy | LOW | Issue CloudFront invalidation for `/favicon.ico` and `/img/*`. Resolves within minutes. |
